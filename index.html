Objective

My goal for this project was to use Ghidra's BSim binary similarity engine to compare two families of IoT malware that are often discussed together because they targeted the same population of embedded devices. Mirai is a malicious DDoS botnet. Hajime is a vigilante worm that infects devices and then hardens them. The starting question was simple: are Mirai and Hajime actually code relatives, or are they only similar because they lived in the same IoT ecosystem and solved the same propagation problems?

I treated the common “Mirai versus Hajime” framing as a hypothesis to test, not as a conclusion. If the two families were source-level sisters, BSim should surface shared functions after unpacking and normalizing the samples. If they were not, then the similarity would be ecological rather than genetic: Hajime would be more like an evolutionary response to Mirai's niche than a fork of Mirai itself.

Either outcome is a real result. If BSim surfaces shared functions, that points to shared source or borrowed code. If it does not, that is evidence of convergent evolution, where two authors arrived at similar behavior without sharing code. The interesting part of the project was learning how to make BSim answer that question honestly, rather than letting it answer an easier question about shared runtime libraries.

Why this matters

Mirai and Hajime both spread across MIPS, ARM, and other embedded architectures. If similarity analysis can link malware behavior across architectures, that is a powerful triage tool. If it cannot, analysts need to know its limits before they trust a result.

Mirai

Mirai is the IoT botnet that knocked large parts of the internet offline in late 2016, including the attacks on the security journalist Brian Krebs, the hosting provider OVH, and the DNS provider Dyn. Antonakakis and a large group of co-authors documented its growth to roughly 600,000 infected devices in their USENIX Security 2017 retrospective [1]. Mirai spreads by scanning the internet for devices with open telnet and trying a built in list of default credentials. Once it has a foothold it reports to a centralized command and control server, which can direct the bots to launch a range of denial of service attacks. The source code was published publicly by a user calling themselves Anna-senpai in 2016, which is why so many variants exist today and why MalwareBazaar's Mirai tag covers a sprawling family rather than one clean lineage [1][2].

Hajime

Hajime, which means "beginning" in Japanese, was first reported by Sam Edwards and Ioannis Profetis at Rapidity Networks in October 2016 [3]. It targets the same kinds of devices as Mirai and spreads the same way, through telnet and weak credentials, but it carries no attack payload. Instead it appears to be the work of a vigilante author who closes the ports commonly abused by Mirai and similar worms, including telnet on port 23 and the TR-064 and TR-069 management ports, effectively locking competitors out of the device [5]. Later measurement work by Herwig and colleagues at NDSS 2019 described Hajime as two cooperating ELF modules, a core implant that handles peer to peer tasks and a separate attack module that handles scanning and propagation, with builds for arm5, arm6, arm7, mipseb, and mipsel [4].

The most distinctive thing about Hajime is its command and control design. Rather than calling home to a server, it joins a decentralized peer to peer network built on BitTorrent's DHT for peer discovery and the uTorrent transport protocol for data transfer, with its traffic encrypted using RC4 [3][6]. That decentralized design makes it far harder to take down than Mirai's server based model.

A note on packing

The original Rapidity report described Hajime's transferred P2P modules as using LZ4 compression, and some samples in the wild have been observed packed with UPX. The samples I analyzed were different again. They were packed with a custom framed LZMA stream, which I had to reverse before any analysis was possible. I treat that as one of my own findings rather than something taken from the literature.

Tools Used

Some of these we used in class, so I describe them briefly. The ones we did not cover together (BSim, telfhash and tlsh, qemu user mode, and the custom decoder) I describe in more depth in the walkthrough.

Ghidra

Disassembly and decompilation of every sample. Primary workbench.

BSim

Ghidra's similarity engine. Builds a database of function feature vectors from P-code and queries it for near matches across binaries and architectures.

MalwareBazaar API

abuse.ch sample repository. Queried by hash, telfhash, tlsh, and signature to source and cluster samples.

telfhash / tlsh

Locality sensitive hashes for ELF files. Used to cluster samples related to a ground truth seed.

qemu-user (mips)

MIPS user mode emulation, used while attempting to run the Hajime unpacker under a debugger.

gdb-multiarch

Remote debugging of the emulated MIPS samples through qemu's gdb stub.

Docker

Disposable, network isolated container for safely handling live malware.

Python 3 + pyzipper

Custom reference LZMA1 decoder, ELF wrapper, and the AES zip handling for MalwareBazaar downloads.

Process Walkthrough (step by step)

This is written so a fresh reverse engineer could reproduce it. The work breaks into four phases: building a clean Mirai cohort, getting Hajime into an analyzable state, building the BSim database, and querying it sensibly.

Phase 1: Build a clean Mirai cohort from one ground truth sample

As I explain in the Dead Ends section, querying a broad "Mirai" pile gave misleading matches. The fix was to anchor on a single not stripped sample whose symbols name the functions I care about, then gather only its close relatives.

  1. Find a not stripped seed Run file over your samples and look for one that says not stripped. A stripped binary hides function names. A not stripped one keeps its symbol table, which is what makes it usable as ground truth. 
    file *.elf | grep "not stripped"
    My seed was 0ea04179b34505024e111f254542d43f53765dc03f784e1efd10a88893e52662, named iran.mipsrouter, an ELF 32 bit MSB MIPS-I statically linked binary.
  2. Get a MalwareBazaar API key Create a free account at abuse.ch, generate an Auth-Key, and export it so the script can read it. 
    export MB_API_KEY="YOUR-AUTH-KEY-HERE"
  3. Pull the seed's fuzzy hashes A get_info query returns the seed's telfhash and tlsh. telfhash is a locality sensitive hash built from an ELF's symbol and function layout, and tlsh is a more general fuzzy hash. Samples built from the same source tend to share close values.
  4. Query for relatives three ways My script find_groundtruth_similar.py queries by get_telfhash, get_tlsh, and, as a broad fallback, the Mirai signature filtered to a filename cluster. It deduplicates by SHA256 and ranks every candidate by closeness to the seed: 
    def candidate_score(item, seed_info):
    score = 0
    if arch == "mips": score += 20
    if ftype == "elf": score += 20
    if sig == "mirai": score += 10
    if name == seed_name: score += 50
    if "mipsrouter" in name: score += 40
    if "iran" in name: score += 30
    if "mipsel" in name: score -= 100   # wrong endianness, reject
    # plus a bonus for file sizes close to the seed
  5. Filter to the exact architecture For a tight cohort I kept only binaries that match the seed exactly, checked against file: 
    def is_target_file(file_output):
    s = file_output.lower()
    return ("elf 32-bit msb executable" in s
            and "mips" in s and "mips-i" in s
            and "statically linked" in s)
  6. Download, extract, and keep the matches MalwareBazaar serves samples as AES encrypted zips with the password infected. The script downloads each candidate, extracts it with pyzipper, runs file, keeps only the architecture matches, logs every decision to a CSV, and stops once it has enough. 
    python3 find_groundtruth_similar.py --want 15
Safety

These are live botnet samples. They stay zipped with the infected password until needed, are never executed during sourcing, and analysis happens only inside Ghidra or a network isolated container.

Script output: ranked candidates and KEEP / Reject decisions

Phase 2: Get Hajime analyzable (unpack the LZMA payload)

Every Mirai sample decompiled immediately. The Hajime samples are packed, and unpacking them was the largest single piece of work in this project. The full play by play is in the Dead Ends section. The reproducible steps are:

  1. Confirm the packing In Ghidra the entry at 0x112fe8 is unreachable, the segment is flagged truncated, and the mapped bytes are high entropy.
  2. Identify the stub Disassembled big endian, the entry is an LZMA decompressor: a position independent prologue, the cacheflush syscall (4147), and a range coder with LZMA's constants.
  3. Decode statically on the host A reference LZMA1 decoder reproduces the stub and only reads and decompresses, so it is host safe. 
    python3 lzma_unpack.py
# [+] HajimeN.elf: off=0x102 lc=3 lp=0 pb=2 ... -> HajimeN_unpacked.bin
    Nine of ten samples decoded cleanly, each with its stream at offset 0x102.
  4. Wrap as ELF 
    python3 wrap_elf.py
# [+] HajimeN_unpacked.bin -> HajimeN_unpacked.elf  entry=0x400018
  5. Import into Ghidra File > Import each _unpacked.elf; it auto detects as ELF / MIPS:BE:32 and the entry decompiles.
Ghidra: packed entry with no code, beside the unpacked, decompiling entry

Phase 3: Build the BSim database

BSim stores function feature vectors in a database and matches new functions against them. I used the H2 backend, a single local file with no server. The key choice is the database template.

  1. Analyze every program Import and auto analyze all Mirai samples, the unpacked Hajime ELFs, and the not stripped ground truth into one Ghidra project, and save.
  2. Create the database with the nosize template From a shell in the Ghidra install (exit the GUI first, since H2 allows one process at a time): 
    cd <ghidra_install_dir>/support
./bsim createdatabase file:/<db_dir>/mirai_hajime medium_nosize
    The nosize template ignores varnode size differences of four bytes and up, which is what allows 32 and 64 bit, and cross architecture, code to match. That is deliberate, since it sets up the architecture agnostic comparison.
  3. Generate signatures 
    mkdir ~/bsim_sigs
./bsim generatesigs ghidra:/<project_dir>/MiraiHajime ~/bsim_sigs \
    --bsim file:/<db_dir>/mirai_hajime
  4. Commit signatures 
    ./bsim commitsigs file:/<db_dir>/mirai_hajime ~/bsim_sigs
  5. Connect from the GUI Reopen Ghidra, then BSim > Manage Servers, add the H2 file, and right click any function to search it against the database.
BSim Manage Servers dialog with the H2 database connected

Phase 4: Query with ground truth, not raw confidence

  1. Sanity check the database Query a known helper like generic_checksum from the ground truth sample. It matched into a stripped sample, proving ingestion and architecture handling were correct.
  2. Ignore the libc noise Discard matches for recv, send, malloc, printf, __uClibc_main, and similar. High hit counts there mean shared runtime, not shared malware.
  3. Query the malware functions Search syn_attack, udpplain_attack, and the random and IP generation routines, and note where they match across the cohort.
  4. Pivot through the call graph Where an attack function does not match directly, anchor on a stable helper that does and follow its callers to locate the candidate in the stripped binary.
BSim query results: a named ground-truth function matching into a stripped sample

Dead Ends and How I Got Past Them

Dead End 1: Hajime was packed and Ghidra could not see the code

The first time I loaded a Hajime sample into Ghidra I had no decompiler view at all. The ELF header pointed the entry at 0x112fe8, but when I tried to jump there Ghidra reported "No results for 0x112fe8." The loadable segment was flagged as truncated, and the bytes Ghidra did map looked like high entropy noise instead of instructions. An unreachable entry point plus near random data is the classic signature of a packed binary.

Once I forced the correct big endian setting and disassembled the entry, the picture cleared up. The code at the entry was a decompression stub. It used the standard MIPS position independent prologue to locate itself, called the MIPS cacheflush syscall (number 4147) after writing to memory, and ran a range coder with the exact constants that define LZMA. Hajime ships its real payload as an LZMA compressed blob with a small unpacker in front of it. The only nonstandard part was the property header, which packs the LZMA parameters into two custom bytes instead of the usual single byte plus dictionary size. That custom framing is exactly why tools like xz and Detect It Easy could not name the packer.

What I tried first

My instinct was to let the sample unpack itself and dump the result, inside a throwaway Docker container with networking disabled and capabilities dropped, under qemu user mode emulation with a gdb stub.

Safety

These are live IoT botnet binaries. I never ran them on my host. Everything that involved execution happened inside a network isolated container, and the breakpoint was always set before any malicious code could run.

This failed the same way every time. As soon as the decompressor started, qemu raised a SIGBUS. The reason is specific to MIPS. The unpacker performs unaligned memory accesses, which a real MIPS Linux kernel silently fixes up in its exception handler. qemu user mode does not emulate that fixup, so it faults instead. Full system emulation would have run it, but that meant a kernel, a root filesystem, and moving samples in and out of a virtual machine, which was a lot of overhead for ten files. I also tried Python's built in lzma module, which refused the stream because it has no end marker and because liblzma caps the literal context parameters in a way the LZMA SDK does not.

What actually worked

Since the stub was decoding an ordinary LZMA stream, I wrote my own reference LZMA1 decoder in Python with no parameter cap, so it would decode exactly what the on device stub decodes. It only reads bytes and decompresses, so it was safe to run on my host. It cracked nine of my ten samples, each with its stream at offset 0x102 and standard parameters, every decode ending cleanly on input exhaustion. After wrapping the output as an ELF, the entry finally decompiled:

/* WARNING: Globals starting with '_' overlap smaller symbols at the same address */

void processEntry entry(void)
{
  (*(code *)(_DAT_0047458c + 0x21c))();
  (*(code *)(_DAT_00474588 + 0x1a00))();
  return;
}

The two indirect calls go through gp relative pointers, which is normal for position independent code. The real targets are filled in at runtime through the global offset table, so Ghidra shows the dispatch but cannot resolve the destinations statically. That is fine for BSim, which matches on the structure of decompiled functions rather than on absolute addresses.

Lesson

Hajime ships packed and Mirai does not, and that difference is itself a finding. Static decompression beat dynamic emulation here. Once I recognized the stub as LZMA, reimplementing the decoder was more reliable than fighting qemu.

Dead End 2: BSim was matching the runtime, not the malware

Early on I assumed that any high confidence BSim result would help me identify Mirai behavior. That assumption was wrong. Because my samples were statically linked ELF binaries, the database filled up with shared libc and uClibc code. Functions like recv, send, select, malloc, printf, vfscanf, vfprintf, strlen, strcmp, and __uClibc_main produced a lot of matches, but those matches mostly proved that the binaries shared common runtime code, not that they shared anything specific to Mirai.

This became obvious once I found a sample that was not stripped and used it as ground truth. Its symbol table exposed the malware relevant names I actually cared about, including main, syn_attack, udpplain_attack, daemonize, cleanup_old_processes, and generic_checksum. A second not stripped sample exposed vseattack, makeRandomStr, getRandomIP, init_rand, rand_cmwc, and mainCommSock. When I searched for syn_attack and udpplain_attack against the broad set, though, the only match I usually got was the ground truth binary itself.

At first I read this as a BSim failure, but a smaller test showed the database was working fine. The generic_checksum helper did match into a stripped Mirai sample, which ruled out any problem with ingestion or my architecture setup. The real problem was that my sample set was too broad. MalwareBazaar's Mirai signature pulls in many related Linux IoT botnet variants whose attack modules were not necessarily compiled from the same source. Stable helper routines matched well across variants, while higher level attack routines did not.

That changed my methodology, and it is what drove the ground truth cohort sourcing in Phase 1. Once I rebuilt the comparison against the tight iran.mipsrouter cohort instead of the broad pile, the attack functions started matching across the whole set, as shown in the next section.

Lesson

BSim is not a magic find Mirai button. For statically linked malware, high hit counts can be misleading because they often point at runtime code. The strategy that worked was to use not stripped samples as ground truth, ignore the libc heavy results, and compare within a tight, source aligned cohort rather than across every Mirai tagged binary.

Most Interesting Functions

I reverse engineered three Mirai functions in detail from the not stripped ground truth sample. These show the three things Mirai needs to be a weapon: it has to flood, it has to flood efficiently, and it has to keep the device to itself.

syn_attack — raw socket TCP SYN flood mirai

This is Mirai's classic SYN flood. It opens a raw socket and turns on IP_HDRINCL so it can hand build every IP and TCP header itself, which is what lets it spoof source addresses:

local_2c = socket(2,3,6);                 // AF_INET, SOCK_RAW, IPPROTO_TCP
local_650 = 1;
setsockopt(local_2c,0,3,&local_650,4);    // IP_HDRINCL: I build the headers

The attack options come from a configuration struct passed in param_1, and mapping the offsets the function reads recovers Mirai's TCP option layout: the on/off flag at +0x14, the duration at +0x10, the destination address at +0x20 with its CIDR prefix at +0x24 (a value of 0x20 means a single /32 host, anything smaller means spray the subnet), the source port at +0x1c (randomized when zero), the static payload at +0x25 with its length at +0xa8, and the pacing window at +0xac and +0xb0. Packet size is clamped between 0x28 (40 bytes, the minimum IP plus TCP header) and 0x5dc (1500, the MTU).

It hand builds the IP header (the leading byte 0x45 is IP version 4 with a five word header, 0x40 is a TTL of 64, and 0x06 is protocol TCP) and sets the TCP data offset and control flags so that every packet is a SYN segment. When a netmask is set it derives the network base, the first usable address, and the last address from ip & mask and walks the range, so a single call can flood an entire block. For every packet it randomizes the fields that should look unique (source port, sequence number, IP id, and any random payload), recomputes both checksums, and sends:

local_622 = generic_checksum(&local_62c,0x14);             // IP header checksum
local_608 = tcp_udp_checksum(&local_618,uVar4-0x14,...);   // TCP checksum
sendto(local_2c,&local_62c,uVar4,0x4000,&local_63c,0x10);

It loops until the configured duration elapses (it compares time() against a deadline) or the attack flag at +0x14 is cleared. The optional usleep paces the flood when a rate limit is set.

BSim: does this generalize across the cohort?

I treated my reverse engineered syn_attack as the reference sample and queried it against the ground truth aligned cohort. The same function is present in all ten samples. Most matched at a perfect 1.000 similarity, which is expected for statically linked sibling builds compiled from the same source, with one sample coming in lower at 0.896, most likely a slightly different build or compiler settings. That single outlier is what makes the result credible rather than suspicious: identical source compiled the same way should match at or near 1.000.

udpplain_attack — high rate UDP flood mirai

Where syn_attack is built for spoofing, udpplain_attack is built for raw throughput. It opens an ordinary UDP socket rather than a raw one, so the kernel fills in the IP and UDP headers and the function never touches IP_HDRINCL or the checksum helpers. That is the whole point of the "plain" variant: dropping header crafting and source spoofing buys the highest packet rate of any Mirai vector.

__fd = socket(2,1,0x11);                       // UDP socket
do { lVar4 = rand(); *puVar6 = (char)lVar4; }  // random payload bytes
    while (sVar13 != sVar9);
sendto(__fd,local_38,sVar13,0x4000,param_1,0x10);   // blast the target

Its options are overlaid on the incoming sockaddr array rather than a flat struct, but they mirror the SYN flood: a packet size range (clamped again at 0x5dc), a duration, a pacing window, an optional static payload copied in with memcpy, and the same single host versus subnet logic keyed off a prefix byte. The inner loop just randomizes a payload and calls sendto as fast as it can until the clock runs out.

BSim: does this generalize across the cohort?

Querying my reverse engineered udpplain_attack against the cohort gives the same shape of result as the SYN flood: a match in all ten samples, nine at a perfect 1.000 and the same lone outlier, SimilarMirai8, at 0.980. The confidence here (about 410) is a little lower than the SYN flood's (about 540) because the function is slightly smaller. That is worth knowing on its own: BSim confidence scales with function size, so two equally perfect matches can carry different significance.

cleanup_old_processes — killer and honeypot check mirai

This function is part of Mirai's "killer" behavior, the code that keeps the device to itself and watches for analysis sandboxes. It first checks for /proc, and if the directory is missing, which is normal on a crude sandbox but not on a real Linux device, it writes a taunt to stderr that also signs the build:

if (access("/proc",0) != 0)
    fwrite("Looks like you are a honeypot, this tool was made by t.me/flylegit!\n", ...);

That string is a useful indicator of compromise and attributes this variant to a "flylegit" operator. It then walks /proc. The detail I found most telling is how it decides which entries to process: it indexes the C library's __ctype_b character classification table to test whether the first character of each directory name is a digit, so it only looks at numeric PID directories and skips everything else. For each PID it opens /proc/<pid>/comm, reads the process name, and normalizes it by cutting at the first carriage return or newline with strcspn, then stripping trailing whitespace using the same __ctype_b table for the space class:

if ((*(ushort *)((uint)(byte)pdVar3->d_name[0] * 2 + __ctype_b) & 8) == 0) break; // isdigit
snprintf(acStack_118,0x100,"/proc/%s/comm",pdVar3->d_name);
sVar4 = read(open(acStack_118,0),__s,0x3f);   // read each process's command name

In full Mirai this normalized name feeds a blacklist comparison that terminates competing bots and known bad processes. The recovered routine is the enumeration and name normalization core of that logic, plus the honeypot detection, which makes this both an anti competitor and an anti analysis function.

BSim: does this generalize across the cohort?

This is the cleanest match of the three. cleanup_old_processes matches all ten samples at a perfect 1.000 with no outlier at all, SimilarMirai8 included. The confidence is much lower (about 128) because the function is small, so even a perfect match is less individually significant than the larger attack functions. The interesting part is the contrast: SimilarMirai8 diverged from the rest in both attack functions yet is equivalent here. Its differences live in the attack code, not the survival code.

Hajime functions

Hajime was harder to analyze than Mirai because the useful code was hidden behind the LZMA unpacker and the unpacked binaries were stripped. Mirai gave me names like syn_attack because I found a not stripped ground truth sample. Hajime did not. For Hajime, I had to rank the unpacked functions by BSim match count and confidence, then read the actual decompiler output and assign analyst names based on behavior.

The three most important Hajime functions I recovered were hajime_protocol_event_loop, hajime_peer_session_manager, and hajime_peer_table_update. The names are mine, but the code snippets below are from the actual functions. Hajime6 is not listed in the BSim tables because it was the one sample my LZMA unpacking pipeline did not successfully recover into an analyzable ELF.

hajime_protocol_event_loop — P2P message parser and maintenance loop hajime

This is the strongest Hajime function I recovered. It looks like the central protocol parser and scheduler for the unpacked Hajime payload. The first clue is the input validation. The function calculates the end of the incoming buffer with param_1 + param_2, then rejects the message if the final byte is not a null terminator. That is not the shape of a flood routine like Mirai's syn_attack; this is a function that expects a structured peer-to-peer message and refuses malformed input before doing any deeper parsing.

pcVar25 = (char *)(param_1 + param_2);
if (*pcVar25 != '\0') {
    puVar6 = (undefined4 *)(*_DAT_004747e8)();
    *puVar6 = 0x16;
    return 0xffffffff;
}

After the null-terminated buffer check, the function searches the message for markers from Hajime's string table, parses numeric length fields, bounds-checks those lengths, and copies peer data into stack buffers. The most useful evidence is the section that accepts either six-byte or eighteen-byte records. A six-byte record lines up with an IPv4 address plus a two-byte port. An eighteen-byte record lines up with an IPv6 address plus a two-byte port. That makes this look like peer contact parsing, which matches Hajime's decentralized P2P design.

if (iVar9 == 6) {
    if (local_38 + 6 < 0x801) {
        *(undefined4 *)((int)&uStack_1040 + local_38) =
            *(undefined4 *)(local_1720[0] + 1);
        acStack_103c[local_38] = local_1720[0][5];
        acStack_103c[local_38 + 1] = local_1720[0][6];
        local_38 = local_38 + 6;
    }
}
else if ((iVar9 == 0x12) && (local_3c + 0x12 < 0x801)) {
    *(undefined4 *)((int)auStack_840 + local_3c) =
        *(undefined4 *)(local_1720[0] + 1);
    /* copies the rest of the 18-byte IPv6+port record */
    local_3c = local_3c + 0x12;
}

The most important block is the message dispatch. After parsing the message, the function identifies one of five message types and jumps through a table at _DAT_00474588 + 0x1fa0. This is why I labeled it an event loop. It does not just decode one field; it validates a message, extracts peer material, chooses a handler, and transfers control to the appropriate protocol routine.

if (((local_1724 != 0) || (iVar9 != 5)) && (iVar9 - 1U < 5)) {
    uVar19 = (*(code *)(&_gp_1 +
              *(int *)((iVar9 - 1U) * 4 + _DAT_00474588 + 0x1fa0)))();
    return uVar19;
}

The bottom half of the same function handles maintenance. It walks linked lists of peer structures, removes stale entries, frees empty records, schedules future peer activity with random jitter, and writes the next timeout back through param_5. That is the biggest difference from Mirai. Mirai's most interesting functions are weaponized packet builders; Hajime's most interesting function is a peer-to-peer message loop that keeps a decentralized network alive.

if (*(int *)(_DAT_00474584 + 22000) < *(int *)(_DAT_00474584 + 0x50a8)) {
    *param_5 = *(int *)(_DAT_00474584 + 0x50a8) -
               *(int *)(_DAT_00474584 + 22000);
}
else {

The important Hajime functions are not flood routines like Mirai's.
